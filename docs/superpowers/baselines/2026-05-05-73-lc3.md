# RED Baseline: LC3 — Silent layering audit-log

## What this baseline captures

LC3 risk: without a mandatory, operator-visible audit-log line at delegation time,
Team Lead silently layers project deltas onto the shipped agent configuration. The
operator has no way to know that a custom model or system-prompt append was applied.
This mirrors the discipline established by R14/R26 against silent inheritance.

This baseline captures both transcripts — the failure mode WITHOUT the audit-log rule,
and the expected behavior WITH it — before the GREEN rule lands in T4.1.

---

## Fixture: consuming project delta file

**`docs/superpowers/brainstormer.md` in a consuming project:**

```markdown
---
agent: brainstormer
---

## Model

sonnet

## System prompt append

When authoring design docs, use the project's internal template at docs/templates/design.md
rather than the superpowers default structure.
```

---

## Transcript A: WITHOUT LC3 (failure mode — silent layering)

Team Lead detects the delta file, merges the model override (`opus` → `sonnet`) and
the system-prompt append, then dispatches the Brainstormer. No audit line is emitted.

**Operator-visible output (chat surface):**

```text
Dispatching Brainstormer to research and design the solution...
```

The operator sees only the delegation message. There is no indication that:

- The Brainstormer is running on `sonnet` instead of the shipped `opus` default.
- A project-specific template instruction was appended to the system prompt.

If the Brainstormer produces an unexpected design structure or uses a lower-capability
model than expected, the operator has no audit trail to diagnose what changed.

**Consequences of silent layering:**

1. Debugging is opaque — operator cannot distinguish between model-version behavior
   differences and instruction differences.
2. Security gap — a malicious or misconfigured delta is applied without any indicator
   that it ran.
3. Reproducibility failure — two runs on the same issue with different delta files
   produce different behavior with no observable signal.

---

## Transcript B: WITH LC3 (mandatory audit-log emission)

Team Lead detects the delta file, runs the denylist lint (no match), merges the
model and append, then emits the mandatory audit line BEFORE dispatching.

**Operator-visible output (chat surface):**

```text
superteam delta applied: brainstormer (model, system-prompt-append); non-negotiable-rules-sha=a1b2c3d4

Dispatching Brainstormer to research and design the solution...
```

The operator sees:

- Which role was affected: `brainstormer`
- Which fields were applied: `(model, system-prompt-append)`
- That the non-negotiable rules block was read and its SHA was logged: `non-negotiable-rules-sha=a1b2c3d4`

**Stderr fallback (when chat surface unavailable, e.g. headless CI):**

When no chat surface is available, the audit line goes to stderr:

```text
[stderr] superteam delta applied: brainstormer (model, system-prompt-append); non-negotiable-rules-sha=a1b2c3d4
```

Per N9, the line is re-emitted at the start of Team Lead's next chat-bearing message
so the operator never permanently loses the audit trail.

---

## Side-by-side comparison

| Dimension | WITHOUT LC3 | WITH LC3 |
|---|---|---|
| Chat output | Delegation message only | Audit line before delegation message |
| Model applied | `sonnet` (silent) | `sonnet` (logged: `model` in applied fields) |
| Append applied | Yes (silent) | Yes (logged: `system-prompt-append` in applied fields) |
| Rules SHA logged | No | Yes: `non-negotiable-rules-sha=<8-char-prefix>` |
| Operator awareness | None | Full: what was applied, which role, SHA prefix |
| Headless CI behavior | Silent always | stderr + re-emit on next chat message |

---

## Conclusion

LC3 mandates: every delegation where a delta was applied (or explicitly empty/orphaned)
MUST emit a structured audit line on the operator-facing chat surface. Stderr is only a
fallback when no chat surface is available, and re-emit is required on the next
chat-bearing message.

GREEN task that lands AFTER this baseline:

- T4.1: `resolve_role_config` audit-log emission rules (chat-first, stderr fallback per N9)
